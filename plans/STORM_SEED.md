## Decisions Made

- **Central manifest** at `~/.pathly/manifest.json` tracking all hosts (claude, codex, copilot) in one file. Written atomically (.tmp Ôćĺ rename). Backed up before every `--apply`.
- **Skills use same YAML pattern as agents.** Each adapter has `_meta/<skill>_skill.yaml` controlling: `skill` (body source), `destination`, `filename`, `strip_frontmatter`, `wrapper`. `stitch_skill()` added alongside existing `stitch_agent()`.
- **install.yaml per adapter gains a `skills:` section** with `destination` and `structure` (folder vs flat).
- **5 core commands**: `start`, `go` (alias: `continue`), `end`, `help`, `pause`.
- **`start` wakes Director**, not PO directly. Director asks: storm / PO / straight to plan / import PRD.
- **`start`, `go`, `end`, `help` are full skills** (unique logic). **`pause` is an alias** (3 lines: write PAUSED, print resume instructions).
- **`end` retro is optional in manual flow** ("Write a retro? y/n"), **mandatory/auto in team-flow**. Retro is never a blocker.
- **"Remove non-Pathly skills? (y/n)"** prompt during `--apply` when non-owned skills detected. "n" = additive/coexist (Pathly installs its skills, foreign skills untouched). "y" = delete everything not in manifest, then install.
- **`caveman`, `compact-in-chat`, `feature-flow` are NOT Pathly skills** ÔÇö Claude Code harness utilities, never included in Pathly install.
- **Unified prefix in YAML config**: `invocation` and `natural_language` fields per adapter skill YAML. Copilot placeholder `#pathly` until spec confirmed.

## Options Rejected

- **Per-host manifests** (`~/.claude/.pathly-manifest.json` etc.) ÔÇö rejected because cross-host status is impossible, manifest lost if host dir nuked.
- **Alias-only approach for `start`/`end`/`help`** ÔÇö rejected because each has 40+ lines of unique logic that can't be delegated to a one-liner.
- **Full skills for `pause`** ÔÇö rejected because pause logic is 3 lines; alias to `pathly.md` is sufficient.
- **Deferring unified prefix to a follow-on feature** ÔÇö rejected because adapter skill YAMLs must define invocation at creation time.

## Constraints

- `pathly-setup` installs both agents AND skills ÔÇö one command, full surface.
- Skills install must be additive by default (no surprise deletions without explicit `--clean` or "y" prompt).
- Platform prefix is config only (YAML fields) ÔÇö no new runtime abstraction layer.
- `core/skills/` stays lean ÔÇö aliases don't get their own `.md` files.
- Codex plugin.json paths must be fixed to match new `pathly-adapters/` structure.

## Context Summary

This feature extends `pathly-setup` to install skills (not just agents) using the same YAML-driven stitch pattern already used for agents. It also defines 5 canonical commands (start, go, end, help, pause) with Director as the router for `start`, and establishes a unified invocation prefix per platform (slash/at/hash) as YAML config in each adapter's skill metadata. A central manifest at `~/.pathly/manifest.json` tracks ownership across all hosts.
